# NoCode 版本管理 V2 设计草案（2026-04-20）

## 1. 背景

当前 NoCode 项目已经具备版本管理第一阶段能力：

- 用户手动提交版本
- 版本列表
- 版本详情
- 版本备注
- 回滚到指定版本
- 真实前端文件版本目录 + 数据库索引
- 回滚生成新的版本记录

第一阶段已经明确不做：

- 面向用户的代码 diff
- 自动推断“这次改了什么”
- 开发者式 patch / 文件变更展示

这套能力已经能完成“可靠存档、可靠查看、可靠回滚”，但从产品体验上仍更接近“可用的数据列表”，距离成熟 NoCode 产品中的“版本管理中心”还有一段差距。

V2 的目标不是重做底层，而是在现有版本快照模型上，进一步补齐更强的产品语义、状态表达和历史体验。

---

## 2. V2 目标

V2 版本管理的目标是：

**把当前版本系统从“版本快照列表”升级为“面向 NoCode 用户可理解、可操作、可回溯的版本管理中心”。**

核心强调：

1. 用户能快速理解当前版本关系
2. 用户知道哪一版正在使用
3. 用户知道某一版是不是稳定版本
4. 用户知道某一版是不是回滚生成
5. 用户能在不理解代码的前提下完成版本回溯与恢复
6. 用户看到的是“版本历史”，不是“代码差异”

---

## 3. 产品原则

### 3.1 面向 NoCode 用户
版本管理展示必须是产品语义，而不是开发语义。

应该展示：
- 当前版本
- 稳定版本
- 回滚生成
- 创建时间
- 备注
- 来源
- 从哪一版恢复而来

不应该展示：
- 代码 patch
- 文件 diff
- 复杂技术术语
- 自动编造的“变更摘要”

### 3.2 快照优先
版本的本质仍然是“某一时刻可恢复的前端结果快照”。

### 3.3 回滚不是覆盖历史，而是生成新历史
回滚操作必须保留痕迹，生成新的版本记录，避免破坏历史链。

### 3.4 显式语义优先于推断
例如 `sourceType=ROLLBACK` 这类来源字段，应显式赋值，不依赖标题文案推断。

---

## 4. 当前已有能力（V1）

### 4.1 后端
- 版本提交接口
- 版本列表接口
- 版本详情接口
- 回滚接口
- 版本文件目录落盘
- 数据库存储版本索引
- `sourceType` 已支持 `MANUAL_COMMIT / ROLLBACK`

### 4.2 前端
- 版本管理页面
- 提交版本表单
- 版本列表
- 版本详情
- 版本备注展示
- 回滚入口
- 版本来源文案映射

### 4.3 当前不足
- 版本状态表达还弱
- 缺少“当前使用中 / 稳定版本”这类语义层
- 缺少更明显的时间线/历史流表达
- 回滚记录虽存在，但与来源版本关系表达还不够强
- 产品仍然偏“表单 + 列表”，不够像版本中心

---

## 5. V2 建议补充的核心能力

## 5.1 版本状态（强烈建议优先做）

### 目标
为每条版本记录补充更强的状态语义，让用户一眼看懂“这条记录现在是什么角色”。

### 建议状态/标记
- `CURRENT`：当前正在使用的版本
- `STABLE`：用户主动标记的稳定版本
- `ARCHIVED`：普通历史版本
- `ROLLBACK_RESULT`：由回滚操作生成的新版本

### 说明
状态不一定要做成单一字段枚举，也可以采用：
- `versionStatus` 保留运行态（READY / FAILED）
- 另加一个“展示/业务语义字段”表示版本角色

### 推荐实现
新增字段：
- `versionRole` varchar(32)

建议值：
- `CURRENT`
- `STABLE`
- `NORMAL`
- `ROLLBACK_RESULT`

---

## 5.2 版本关系表达

### 目标
让用户能看懂“这条版本从哪来”。

当前已有：
- `parentVersionNo`

但这还不够区分“普通继承”与“回滚来源”。

### 建议新增字段
- `sourceVersionId` bigint NULL

### 含义
- 手动提交版本时，表示它基于哪一版当前结果提交
- 回滚生成版本时，明确指向“回滚目标版本”

### 价值
前端可以展示：
- 回滚生成，自版本 V6 恢复
- 当前版本继承自 V8

这会让历史更像“时间线”，而不是散列表。

---

## 5.3 当前版本指示

### 目标
版本列表里必须非常明确：哪一版当前正在使用。

### 推荐方案
前端展示层：
- 在列表中给当前版本打 `当前使用中` 标签
- 回滚成功后自动切换当前版本指示

### 后端实现选择
两种方式：

#### 方案 A：从应用表记录当前版本 id
在 `app` 表补：
- `currentVersionId`

优点：
- 查询快
- 当前版本语义非常清晰

缺点：
- 需要同步维护

#### 方案 B：从版本表里用 `versionRole=CURRENT` 表达
优点：
- 数据集中在版本表

缺点：
- 要处理“只有一个 CURRENT”的约束

### 当前推荐
优先选 **A**：
- 在 `app` 表增加 `currentVersionId`
- 版本列表中基于此标记当前版本

这个更稳，也更贴合产品语义。

---

## 5.4 稳定版本标记

### 目标
让用户能把某一版标记成“放心可恢复”的版本。

### 能力形式
- 在版本详情页 / 列表上增加“标记为稳定版本”
- 支持取消标记

### 数据方案
最小做法：
- `versionRole=STABLE`

更稳做法：
- 新增布尔字段 `isStable`

### 当前推荐
新增：
- `isStable` tinyint default 0

原因：
- 比 role 更清晰
- 一个版本可以同时是 `CURRENT + STABLE`
- 不和 CURRENT/ROLLBACK_RESULT 产生冲突

---

## 5.5 版本时间线展示

### 目标
让版本历史的可读性更强。

### 前端建议
版本列表展示增强为：
- 版本号
- 标题/备注
- 创建时间
- 来源标签
- 当前 / 稳定标签
- 若为回滚结果，展示“恢复自 Vx”

### 是否需要专门时间线组件
当前阶段不必须。

可以先做成“增强版列表卡片”，等后续再升级成真正 timeline UI。

---

## 6. V2 数据模型建议

## 6.1 版本表新增字段建议
基于 `app_frontend_version`：

### 建议新增
- `sourceVersionId` bigint NULL COMMENT '来源版本 id，尤其用于回滚来源表达'
- `versionRole` varchar(32) NOT NULL DEFAULT 'NORMAL' COMMENT '版本角色:CURRENT/NORMAL/ROLLBACK_RESULT'
- `isStable` tinyint NOT NULL DEFAULT 0 COMMENT '是否稳定版本'

### 现有字段继续保留
- `versionNo`
- `versionTitle`
- `changeSummary`（版本备注）
- `userPrompt`
- `sourceType`
- `versionPath`
- `metaPath`
- `parentVersionNo`
- `versionStatus`

---

## 6.2 应用表建议新增字段
若当前 `app` 表尚未有当前版本指向，建议新增：

- `currentVersionId` bigint NULL COMMENT '当前使用中的前端版本 id'

### 价值
- 前端版本页能够明确高亮当前版本
- 回滚后能快速更新应用当前版本指向
- 部署、预览、恢复等后续能力也更容易围绕它扩展

---

## 7. V2 前端页面建议

## 7.1 版本列表
建议从“普通列表”升级为“带状态卡片的版本列表”。

每条至少展示：
- 版本号
- 标题
- 备注
- 时间
- 来源（手动提交 / 回滚生成）
- 标签（当前使用中 / 稳定）

若是回滚版本：
- 展示“恢复自 Vx”

## 7.2 版本详情
建议补充：
- 当前是否为当前版本
- 是否为稳定版本
- 来源版本
- 回滚关系说明

## 7.3 操作区
每个版本建议至少支持：
- 查看详情
- 回滚到该版本
- 标记 / 取消稳定版本

---

## 8. 分阶段实现建议

## Phase 2A（最小可落地，建议先做）
目标：把 V1 的版本列表升级成真正可理解的版本中心。

### 范围
- 后端新增 `sourceVersionId`
- 后端新增 `isStable`
- 应用表新增 `currentVersionId`
- 回滚时写入 `sourceVersionId`
- 回滚/提交后更新 `currentVersionId`
- 前端版本列表展示 `当前使用中 / 稳定 / 回滚生成`

### 验证标准
- 用户能一眼看出当前版本
- 用户能看出哪版是稳定版
- 用户能看出某条记录是不是回滚生成
- 回滚后当前版本标识正确切换

---

## Phase 2B（体验增强）
### 范围
- 时间线样式
- 更清晰的回滚来源说明
- 稳定版本筛选/快捷入口

---

## Phase 2C（未来可选）
### 范围
- 视觉对比
- 发布版本标记
- 版本预览快照图
- 开发者视图（若未来真的需要）

---

## 9. 当前最推荐的 Next Action

如果只做一个最小、最值、最贴近 nocode.cn 版本管理体验的子步，推荐：

**先做 `currentVersionId + isStable + sourceVersionId` 这 3 个核心语义字段，并把前端版本列表升级成带“当前使用中 / 稳定 / 回滚生成”标签的版本中心。**

原因：
- 改动面可控
- 用户价值很直观
- 不需要引入代码 diff
- 最符合 NoCode 版本管理心智
- 能明显拉开和“仅仅是版本列表”的差距

---

## 10. 一句话总结

V2 的重点不是把版本功能做成 Git，而是把它做成：

**一个 NoCode 用户看得懂、敢使用、能回溯、能恢复的版本管理中心。**
