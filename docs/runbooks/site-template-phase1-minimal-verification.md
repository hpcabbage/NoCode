# Runbook: site_template 第一阶段最小验证

## 适用场景

用于在 NoCode 当前 `site_template` 第一阶段收口过程中，做一轮不依赖大量人工点页面的最小验证。

目标不是覆盖完整前端验收，而是确认当前后端与接口主链路是否具备基础闭环。

## 前置条件

1. 已启动 NoCode 后端服务；
2. 数据库、Redis 等基础依赖可用；
3. 当前环境存在可登录账号；
4. 当前数据库中已有至少一个可用于“保存为模板”的应用，或允许先创建一个测试应用；
5. 若要验证管理员能力，需要准备管理员账号。

## 步骤

### 步骤 1：确认模板相关接口已注册
优先检查当前代码中是否存在以下核心接口：

- `/template/add`
- `/template/create/from-app`
- `/template/update`
- `/template/get/vo`
- `/template/list/page/vo`
- `/template/admin/update`
- `/template/delete`

说明：
这一步是最小静态核对，用于先确认接口闭环没有在代码层缺件。

### 步骤 2：使用普通用户验证模板创建主链路
按下面顺序验证：

1. 登录普通用户；
2. 准备一个已有应用 id；
3. 调用“从应用创建模板”接口；
4. 记录返回的模板 id；
5. 调用模板详情接口，确认模板可被读取。

期望结果：
- 能成功创建模板；
- 返回有效模板 id；
- 详情查询能读到对应模板。

### 步骤 3：验证模板列表可见性
按下面顺序验证：

1. 使用普通用户请求模板分页列表；
2. 观察返回结果是否符合当前公开/私有可见性约束；
3. 如有条件，可切换不同用户再次查询，确认列表权限收口生效。

期望结果：
- 列表接口可正常返回；
- 非公开模板不会被不该看到的用户随意读到；
- 自己应能看到自己有权限访问的模板。

### 步骤 4：验证模板编辑能力
按下面顺序验证：

1. 使用模板拥有者账号调用模板更新接口；
2. 修改名称、描述、分类或封面等基础字段；
3. 再次调用模板详情接口确认字段变更生效。

期望结果：
- 模板拥有者可成功更新自己的模板；
- 更新后详情查询结果与修改内容一致。

### 步骤 5：验证管理员公开状态控制
按下面顺序验证：

1. 使用管理员账号调用 `/template/admin/update`；
2. 修改模板 `isPublic` 状态；
3. 重新查询模板详情或模板列表，确认可见性变化符合预期。

期望结果：
- 管理员可修改模板公开状态；
- 状态修改后，模板详情 / 列表结果发生对应变化。

### 步骤 6：验证模板删除能力
按下面顺序验证：

1. 使用模板拥有者或管理员调用删除接口；
2. 删除刚才创建的测试模板；
3. 再次查询模板详情，确认记录不可再正常读取。

期望结果：
- 拥有者或管理员可以删除模板；
- 删除后详情接口不再返回有效模板内容。

## 验证方式

最小通过标准：

1. 模板创建、详情、列表、编辑、管理员公开控制、删除，这 6 条主链路都能跑通；
2. 模板权限边界没有明显穿透；
3. 当前结果能支撑“`site_template` 第一阶段后端最小闭环已完成”的表述。

若以上仅完成接口级验证，而未完成页面人工验收，则结论仍应保持为：

- `site_template 第一阶段后端最小闭环已完成`
- `页面级最终验收待补`

## 常见失败与处理

### 1. 创建模板失败
优先检查：
- 应用 id 是否真实存在；
- 当前用户是否有权访问该应用；
- 应用转模板时依赖的字段是否为空。

### 2. 列表结果异常
优先检查：
- 当前登录态是否正确；
- 模板 `isPublic` 状态是否符合预期；
- 列表查询条件是否带入了错误过滤项。

### 3. 编辑失败
优先检查：
- 当前用户是否为模板拥有者；
- 请求体中模板 id 是否正确；
- 必填字段是否缺失。

### 4. 管理员更新失败
优先检查：
- 当前账号是否具备管理员权限；
- 是否走了 `/template/admin/update` 而不是普通更新接口。

### 5. 删除后仍能查到模板
优先检查：
- 是否命中了缓存或旧响应；
- 删除接口是否真正返回成功；
- 查询是否仍在读取旧数据源。
