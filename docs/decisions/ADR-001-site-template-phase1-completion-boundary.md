# ADR-001: site_template 第一阶段完成边界定义

## 状态
- Accepted

## 背景

NoCode 当前真实主线已经切换到 `site_template` 第一阶段收口。

经过前序代码核对与文档整理，已经能够确认：
- 模板实体、服务、控制器与基础 DTO / VO 已存在；
- 模板列表、详情、从应用创建模板、保存模板、编辑模板、管理员控制公开状态等后端能力已具备；
- 文档主入口已经从旧的 HTML streaming 落盘排查，切换到 `site_template` 第一阶段语境。

但当前还存在一个容易反复出现的问题：

**“代码最小闭环已具备”** 和 **“页面级完整正式验收完成”** 不是一回事。

如果不显式定义完成边界，后续会话很容易：
- 要么把当前阶段说得过满；
- 要么反过来误以为这轮还停留在旧问题排查期；
- 进一步导致文档、提交说明和阶段判断持续漂移。

## 决策

对 `site_template` 第一阶段，统一采用下面的完成边界定义：

### 1. 可以认定为“已完成”的部分
以下能力视为第一阶段已经完成的最小闭环：

- 模板基础数据结构已建立；
- 模板列表 / 详情后端能力已具备；
- 支持从应用创建模板；
- 支持用户编辑自己创建的模板；
- 支持管理员控制模板公开状态；
- 模板列表可见性约束已做第一轮收口；
- 文档入口、当前计划与专题规格已经切换并对齐到 `site_template` 主线。

### 2. 不认定为“已完成”的部分
以下内容不计入“第一阶段已完整完成”：

- 页面级完整人工验收；
- 更重的模板运营能力（例如推荐、排序、审核流、精细化后台运营策略）；
- 全量回归 runbook；
- 对外可宣称“正式稳定上线”的完整发布证明。

### 3. 对外描述约定
后续在文档、提交说明、阶段汇报里，统一使用类似表述：

- `site_template 第一阶段后端最小闭环已完成`
- `site_template 第一阶段基础能力已具备，页面级最终验收待补`

避免使用：

- `site_template 已全部完成`
- `模板系统已经完整上线`

除非后续补齐了页面级验收、运行手册和更完整的阶段验证证据。

## 为什么这样做

1. **真实**
   当前代码已经不是“还没做”，但也确实没到“所有验收都做完”的程度。

2. **防止表述失真**
   如果不明确边界，最容易出现的就是阶段完成度被说大或说小，两边都误导。

3. **方便后续提交与协作**
   有了明确边界，后续写 changelog、提交说明、阶段同步时就不容易漂。

4. **适合当前客观条件**
   老板暂时不方便做人点页面验收，所以当前更适合先把“可非人工确认的最小闭环”收口清楚。

## 备选方案

### 方案 A：暂不定义完成边界，等页面全部验收完再说
不采用。

原因：
- 会让当前阶段长期停留在模糊状态；
- 文档和提交说明容易继续漂；
- 后续会话仍然会反复误判主线与完成度。

### 方案 B：直接把第一阶段定义为“完整完成”
不采用。

原因：
- 与当前实际验证证据不匹配；
- 页面级最终人工验收还没有补齐；
- 容易在后面暴露出“纸面完成、实际未验”的问题。

## 影响

采用本决策后：

- `site_template` 第一阶段可以作为一个“后端最小闭环已完成”的阶段成果继续推进；
- 文档、changelog、后续提交说明都应按该边界表述；
- 后续新增 runbook、验收记录或提交说明时，需要继续服从这条边界，不应提前夸大完成度。

## 后续动作

1. 后续可补一份 runbook，沉淀模板能力的最小验证路径；
2. 在有条件时补页面级人工验收记录；
3. 如果模板能力进入第二阶段，再新增对应 spec / decision，而不是继续混写在第一阶段文档里。
